# 健康 RAG 项目部署文档（最小可用版）

本文档对应仓库根目录下的 `docker-compose.yml`，目标是快速完成“可访问、可演示、可复现”的部署闭环。

## 1. 服务器准备

1. 系统：Ubuntu 22.04+（推荐）
2. 安装 Docker 与 Docker Compose 插件
3. 开放端口：`3000`、`8000`、`6333`（如需公网直接访问）

## 2. 拉取代码

```bash
git clone https://github.com/yuanguihuafang/Health-Rag-Assistant
cd Health-Rag-Assistant
```

## 3. 配置环境变量

```bash
cp backend/.env.example backend/.env
```

重点检查以下配置：

1. `LLM_BASE_URL` / `LLM_API_KEY` / `LLM_MODEL`
2. `EMBEDDING_BASE_URL` / `EMBEDDING_API_KEY` / `EMBEDDING_MODEL`
3. `RERANK_BASE_URL` / `RERANK_API_KEY` / `RERANK_MODEL`
4. `QDRANT_URL=http://qdrant:6333`
5. `REDIS_HOST=redis`（如项目中启用 Redis 配置）

## 4. 启动服务

```bash
docker compose up -d
```

查看状态：

```bash
docker compose ps
docker compose logs -f backend
```

## 5. 访问地址

1. 前端：`http://<server-ip>:3000`
2. 后端：`http://<server-ip>:8000`
3. Qdrant：`http://<server-ip>:6333/dashboard`

## 6. 首次数据初始化

```bash
docker compose exec backend python manage.py import_cmedqa_to_kb --limit 3000
docker compose exec backend python manage.py rebuild_health_rag_vectors --progress-step 100
docker compose exec backend python manage.py sync_health_rag_vectors --progress-step 100
```

验证向量库状态：

```bash
docker compose exec backend python manage.py shell -c "from health_rag_assistant.services.vector_store import QdrantVectorStore; print(QdrantVectorStore().status())"
```

## 7. 评估与留档

检索评估：

```bash
docker compose exec backend python manage.py evaluate_health_rag
```

RAGAS 评估：

```bash
docker compose exec backend python manage.py evaluate_health_rag_ragas --input health_eval_questions_cmedqa_v1.jsonl --limit 20 --top-k 5
```

结果文件目录：

`backend/health_rag_assistant/datasets/eval/results/`

## 8. 常见问题

1. `ConnectionError` 访问 embedding/rerank 失败：优先检查服务器出网策略、防火墙、代理配置。
2. `collection 不存在`：先执行 `sync_health_rag_vectors`。
3. 后台新增文档慢：优先确认 embedding 服务可达与限流策略。
4. `manage.py check` 有 `staticfiles.W004`：确认 `backend/static` 目录存在。

## 9. 生产化建议（下一步）

1. 用 `gunicorn/daphne + nginx` 替代 Django `runserver`。
2. 前端改为 `npm run build` 后由 nginx 托管静态文件。
3. 引入 HTTPS、域名、访问控制与日志采集。
4. 索引重建与大文件导入改为 Celery 异步任务。

## 10. 自动部署（GitHub Actions）

仓库已提供工作流：

`/.github/workflows/deploy.yml`

触发方式：

1. push 到 `main` 分支自动触发
2. 在 GitHub Actions 页面手动点 `Run workflow`

### 10.1 服务器准备

在服务器上先安装：

1. `git`
2. `docker` / `docker compose`

并确保目标目录存在写权限，例如：

`/opt/Health-Rag-Assistant`

### 10.2 GitHub Secrets

在仓库 Settings -> Secrets and variables -> Actions 中添加：

1. `DEPLOY_HOST`：服务器公网 IP
2. `DEPLOY_PORT`：SSH 端口（默认 22）
3. `DEPLOY_USER`：SSH 登录用户（如 `root` 或普通 sudo 用户）
4. `DEPLOY_SSH_KEY`：私钥内容（建议专用 deploy key）
5. `DEPLOY_APP_DIR`：服务器部署目录（如 `/opt/Health-Rag-Assistant`）

### 10.3 首次上线步骤

1. 手动在服务器执行一次：
   - `git clone`
   - `cp backend/.env.example backend/.env`
   - 编辑好真实 `backend/.env`（模型 key、Qdrant 等）
2. 在 GitHub 上触发一次 `Deploy To Server` 工作流。
3. 工作流会执行：
   - `git pull` 最新代码
   - `docker compose up -d --build`
   - 健康检查 `GET /api/health-rag/health/`

### 10.4 面试可讲要点

1. 我把部署流程做成了 CI/CD：代码推送后自动拉取、自动构建、自动部署。
2. 通过 GitHub Secrets 管理服务器连接信息，避免密钥入库。
3. 部署后执行健康检查，保证服务可用性可验证。
